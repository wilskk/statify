# Desain Fitur: Komponen & Utilitas

Dokumen ini berisi paket desain (Design Package) untuk Feature Set `Komponen & Utilitas`.

---

## 2. Design Package

### 2.1. Diagram Urutan (Sequence Diagrams)

*Diagram tidak terlalu relevan untuk fitur ini. Fitur ini lebih tentang penyediaan blok bangunan (building blocks) yang dapat digunakan kembali.*

### 2.2. Penyempurnaan Model Objek (Object Model Refinements)

*Perubahan pada model objek (kelas, atribut, metode baru) yang ditemukan selama desain.*

- **Komponen UI Dasar (`@/components/ui/`):**
  - Menggunakan `shadcn/ui` sebagai basis untuk komponen UI yang konsisten dan dapat diakses, seperti `Button`, `Dialog`, `Input`, `Select`, dll. Ini memastikan tampilan dan nuansa yang seragam di seluruh aplikasi.
- **Utilitas Umum (`@/lib/utils.ts`):**
  - Berisi fungsi pembantu murni yang dapat digunakan kembali di mana saja, contohnya adalah `cn` untuk menggabungkan nama kelas `tailwind` secara kondisional.
- **Komponen Umum (`@/components/Common/`):**
  - `iconHelper.tsx`: Sebuah komponen yang memetakan nama ikon ke SVG atau komponen ikon sebenarnya, menyederhanakan penggunaan ikon di seluruh aplikasi.
  - `VariableListManager.tsx`: Komponen stateful yang kompleks dan dapat digunakan kembali, dirancang khusus untuk menangani logika pemilihan variabel (memindahkan variabel antar daftar) yang umum ditemukan di semua dialog analisis.

### 2.3. Catatan Alternatif Desain (Design Alternatives)

*Diskusi dan keputusan mengenai pilihan desain yang signifikan.*

- **Alternatif 1:** Membuat setiap komponen UI dari awal secara manual.
  - **Kelebihan:** Kontrol penuh atas gaya dan perilaku.
  - **Kekurangan:** Memakan waktu sangat lama, sulit untuk memastikan konsistensi dan aksesibilitas (WAI-ARIA).
- **Keputusan:** Mengadopsi `shadcn/ui`. Ini memberikan fondasi yang kuat dari komponen yang indah, dapat disesuaikan, dan dapat diakses, memungkinkan tim untuk fokus pada fungsionalitas aplikasi daripada membangun infrastruktur UI dari nol.

- **Alternatif 2:** Setiap dialog analisis mengimplementasikan logika pemilihan variabelnya sendiri.
  - **Kelebihan:** Tidak ada abstraksi yang dibutuhkan pada awalnya.
  - **Kekurangan:** Akan ada duplikasi kode yang masif di puluhan dialog analisis. Setiap perbaikan atau perubahan harus diterapkan di banyak tempat.
- **Keputusan:** Menciptakan komponen `VariableListManager` yang dapat digunakan kembali. Ini mengenkapsulasi semua logika dan state yang kompleks, sehingga setiap dialog analisis hanya perlu menggunakan komponen ini dan mendapatkan fungsionalitas yang konsisten dan teruji.
